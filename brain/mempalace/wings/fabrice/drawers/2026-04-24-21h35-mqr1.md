### 🔴 BUGS ACTIFS — Les 3 qui causent les désynchros que tu vois

---

**Bug 1 — `mark_cross_published` : inversion fichier primaire / side-effect**

**Fichier :** `backend/jarvis/src/lib/action-executor.ts`
**Ligne :** `resolveFilePath()` case `mark_cross_published`

Le fichier primaire (celui qui est lu + modifié + committé en mode fiable) est `cross-execution-log.md`. C'est correct.

Mais dans `applySideEffects()`, la side-effect 1 appelle `markCrossPublished()` (de `markdown.ts`) sur le `cross-engagement-tracker.md`. Cette fonction cherche dans le tracker une ligne qui contient le texte du post — et ce texte ne matche jamais correctement parce que le tracker a des titres de posts longs et que le paramètre `post` que tu envoies est un libellé court du chat ("M5 Pipeline conversion recap"). **Résultat : la side-effect échoue silencieusement, rien n'est logué, tu ne le vois jamais.**

La vraie ironie : le fichier primaire est le bon (`cross-execution-log.md`), mais la logique de matching dans `applyTransform()` pour ce fichier a le même problème de matching de texte que la side-effect.

**Bug 2 — Le matching de texte par contenu est fondamentalement cassé**

**Fichier :** `action-executor.ts`, fonction `applyTransform()`, case `mark_cross_published`

```typescript
const matchByContent = postLower.length > 5 && cellPost.toLowerCase().includes(postLower.slice(0, 25));
const matchByPartial = cellPost.toLowerCase().includes(postLower) || postLower.includes(cellPost.toLowerCase().slice(0, 20));
```

Le problème : tu passes "M5 Pipeline conversion recap" dans `post`, mais la colonne "Post cible" dans le log contient "R Twitter 18h30 — M5 pipeline recap". Les 25 premiers caractères de "M5 Pipeline conversion recap" = "M5 Pipeline conversion r" — ça ne matche pas "R Twitter 18h30 — M5 p".

**Il n'y a pas de matching par ID.** Pourtant les lignes ont un ID explicite (B6, B7, A12...). C'est la solution évidente qui n'a pas été implémentée.

Fix : ajouter un paramètre `cross_id` dans les params de `mark_cross_published` et matcher directement `cells[1].trim() === crossId`.

**Bug 3 — Cache SHA + batch en rafale = commits fantômes**

**Fichier :** `backend/jarvis/src/lib/cache.ts` + `github.ts`

Le cache a un TTL de 60 secondes. Quand tu valides 3-4 actions en rafale dans l'UI (batch), `action-execute-batch.ts` groupe bien les actions par fichier et chaîne les transforms — c'est correct. Mais quand tu valides les actions **une par une** en cliquant sur chaque bouton Valider, chaque `executeAction()` fait son propre `ghRead()`. Si la première action commit (SHA abc → def) et que la deuxième lit encore le cache (SHA abc), le deuxième commit échoue avec 422 de GitHub. L'erreur est catchée, loggée dans Railway, mais la réponse HTTP renvoie quand même `ok: true` au frontend puisque le catch est dans `applySideEffects`.

**Le plus grave :** dans `action-execute.ts` (la route single), `cacheInvalidateAll()` est appelé APRÈS `executeAction()`. Mais si `executeAction()` a déjà fait un `ghUpdate()` interne qui utilise le cache — le SHA utilisé pour la side-effect qui suit est potentiellement déjà périmé.

### 🟡 PROBLÈMES D'ARCHITECTURE

---

**Problème 4 — `action-execute-batch.ts` n'exécute pas les side-effects**

**Fichier :** `action-execute-batch.ts`

Le batch route groupe les actions par fichier et fait 1 `ghUpdate` par groupe avec un `reduce` sur les transforms — c'est efficace. Mais il n'appelle jamais `applySideEffects()`. Résultat : quand tu valides un batch de marks_cross_published, le `progress-semaine.md` et le `cross-engagement-tracker.md` ne sont jamais mis à jour en side-effect. C'est pourquoi les fix en batch que j'ai proposés ce soir n'ont mis à jour que le fichier principal.

Fix immédiat : après le `ghUpdate` de chaque groupe, appeler `applySideEffects()` pour chaque action du groupe (non-bloquant, même pattern `.catch()`).

---

**Problème 5 — `markCrossPublished()` dans `markdown.ts` n'est plus aligné avec le format réel du fichier**

**Fichier :** `backend/jarvis/src/lib/markdown.ts`

`markCrossPublished()` cherche une ligne qui contient `postSummary` et met à jour `cells[4]` et `cells[5]`. Mais le format actuel du `cross-execution-log.md` a 7 colonnes : `|ID|Jour|Post cible|Heure cible|Heure réelle|Statut|Notes|`. Les index de cellules sont donc 1, 2, 3, 4, 5, 6, 7 — et dans `applyTransform`, le code met à jour `cells[5]` (Heure réelle) et `cells[6]` (Statut), ce qui est correct. Mais `markCrossPublished()` dans `markdown.ts` met à jour `cells[4]` (Heure cible) et `cells[5]` (Heure réelle) — c'est **décalé d'une colonne**.
