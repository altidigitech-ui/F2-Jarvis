**Cause 3 — Cache + SHA en cascade**

Quand je propose 4 actions dans le même message et que tu les valides en rafale, toutes lisent le même SHA de fichier au moment de la proposition. Si deux actions touchent le même fichier, la première commit (SHA: abc → def) mais la deuxième essaie encore avec SHA: abc → GitHub refuse avec 422. Dans les side-effects, `.catch(() => {})` avale l'erreur sans bruit.

## Le fix systémique — ce que tu dois changer dans le code

C'est un bug backend. Voici exactement les 3 changements à faire dans `backend/jarvis/src/lib/action-executor.ts`.

---

**Fix 1 — Supprimer les `.catch(() => {})` qui avalent tout**

Actuellement toutes les side-effects font :
```typescript
await ghUpdate(...).catch(() => {});
```

Remplacer par :
```typescript
await ghUpdate(...).catch((err) => {
  console.error(`[action-executor] side-effect failed for ${actionType}:`, err);
});
```

Ça ne change pas le comportement — les side-effects restent non-bloquantes — mais au moins les erreurs apparaissent dans les logs Railway. Tu sauras quand ça merde.

---

**Fix 2 — `mark_cross_published` cible le MAUVAIS fichier primaire**

Dans `resolveFilePath()`, ligne ~120 :
```typescript
case "mark_cross_published":
  return {
    path: `${persona}/cross-engagement-tracker.md`, // ← WRONG
```

Changer en :
```typescript
case "mark_cross_published":
  return {
    path: `${persona}/engagement/cross-execution-log.md`, // ← PRIMARY
    commitPrefix: `${persona}: cross published`,
  };
```

Puis dans `applyTransform()`, remplacer le case `mark_cross_published` qui appelle `markCrossPublished()` (sur le tracker) par une fonction qui update directement le `cross-execution-log.md` avec un match plus robuste.

---

**Fix 3 — Le matching de texte est fragile**

Dans la side-effect `mark_cross_published` de `applySideEffects()`, le code cherche :
```typescript
lines[i].toLowerCase().includes(post.toLowerCase().slice(0, 30))
```

Mais `post` vaut "REFAIT Pro teaser + pipeline" et la ligne contient "R LinkedIn 21h — Final beta". **Aucun match → rien committé.**

La fix propre : ajouter un paramètre `cross_id` (ex: "B8") dans les params de `mark_cross_published` et matcher par ID dans le log. Si ce n'est pas possible, matcher par jour + heure plutôt que par titre.
