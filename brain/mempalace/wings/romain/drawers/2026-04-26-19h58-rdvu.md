## DIAGNOSTIC — Bug de stream invisible

**Cause racine :** le stream fonctionne côté backend, mais ne traverse jamais jusqu'au navigateur en temps réel. Les messages arrivent bien dans Supabase (tu les vois au refresh) — le problème est purement dans la couche de transport.

---

### **Bug #1 — `writeEvent` ne flush pas** 🔴 (cause principale)

`backend/jarvis/src/routes/chat.ts`, ligne 353 :

```typescript
function writeEvent(res: Response, event: ...) {
  res.write(JSON.stringify(event) + "\n");
  // ← RIEN. Pas de flush.
}
```

Node.js bufferise les writes TCP. Sans `res.flush()`, les petits chunks (pings, tool_use, textes courts) restent dans le buffer Node.js et ne sont pas envoyés immédiatement. Le navigateur attend que le buffer se remplisse — avec de longues sessions à 20-30 appels tools, ça peut prendre plusieurs minutes avant que le premier chunk parte.

---

### **Bug #2 — Header `X-Accel-Buffering: no` absent** 🔴 (cause principale)

`backend/jarvis/src/routes/chat.ts`, lignes 507-511 :

```typescript
res.setHeader("Content-Type", "application/x-ndjson");
res.setHeader("Cache-Control", "no-cache, no-transform");
// ← MANQUE : res.setHeader("X-Accel-Buffering", "no");
res.flushHeaders();
```

Railway utilise nginx comme reverse proxy. Par défaut nginx **bufferise toute la réponse** avant de la transmettre à Vercel. Sans `X-Accel-Buffering: no`, le stream est retenu côté Railway jusqu'à la fin — Vercel ne reçoit rien, le navigateur ne reçoit rien. C'est exactement le symptôme : rien pendant 2-3 min, puis tout d'un coup au refresh.
